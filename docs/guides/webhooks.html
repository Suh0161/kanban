Webhooks

Webhooks turn Elevate into a push-based system. Instead of polling our API every minute to find out what changed, you give us a URL and we'll send a signed HTTP POST every time something interesting happens — a task moves, a comment lands, an attachment uploads. It's how you wire Elevate into Slack, Discord, GitHub Actions, or your own infrastructure with near-zero latency.

Available events

You can subscribe to any combination of these events. Most integrations start by listening for a couple and expand from there:

  • task.created — a new task lands on a board
  • task.updated — a task's title, priority, assignee, tags, due date, or description changed
  • task.moved — a task switched columns (this is the big one for status-tracking integrations)
  • task.archived — a task got archived but is still recoverable
  • task.restored — a previously archived task came back
  • task.deleted — a task was permanently removed and can't be recovered
  • task.commented — someone added a comment to a task

Create a webhook

The friendly path is Settings → Webhooks in the Elevate app, where you can paste a URL, pick events from a checklist, and save. If you'd rather automate it (one-click setup for self-hosted users, for example), you can do the same thing through the API:

POST/workspaces/{id}/webhooks
curl -X POST https://app.arcnvd.com/api/v1/workspaces/trust-and-safety/webhooks \
  -H "X-API-Key: Elevate_abc123def456" \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com/webhook","events":"task.created,task.updated","active":true}'

The response includes a secret field. This is what you'll use to verify that incoming webhook payloads actually came from Elevate and not some random attacker on the internet who guessed your URL.

The signing secret is shown once — write it down.Once you close that response we throw the raw value away. If you lose it the only path forward is deleting the webhook and creating a new one. Stash it in your secrets manager the moment it lands.

Payload format

When something happens, we send a JSON POST to the URL you registered. The body always includes a top-level event string, the workspace it came from, an ISO timestamp, the webhook ID for debugging, and a data object with the relevant entity:

POSTyour-endpoint
json 
{
  "event": "task.created",
  "workspaceId": "trust-and-safety",
  "timestamp": "2026-05-16T10:30:00.000Z",
  "webhookId": "wh_abc123",
  "data": {
    "task": {
      "id": "task-1",
      "title": "Fix login bug",
      "priority": "High",
      "columnId": "col-1",
      "assigneeId": "user-1",
      "tags": ["bug", "auth"],
      "created_at": "2026-05-16T10:30:00.000Z"
    }
  }
}

HTTP headers

Every webhook request also carries these headers, which together let you verify, route, and deduplicate:

  • X-Elevate-Event — the event type, e.g. task.created. Useful for routing the request to the right handler without parsing the body first.
  • X-Elevate-Signature — an HMAC-SHA256 hex digest of the request body, computed with your secret. This is what you check to confirm the payload is genuine.
  • X-Elevate-Delivery — a unique ID per delivery attempt. Stays the same across retries of the same event, so you can deduplicate.
  • Content-Type: application/json — always.

Signature verification

This is the part you can't skip. Without verification, anyone who guesses or scrapes your webhook URL can send fake payloads and trick your integration into doing things. Verification takes about ten lines of code and shuts that off completely.

The recipe is the same in every language: take the raw request body, HMAC-SHA256 it with your webhook secret, and compare the result to the X-Elevate-Signature header using a constant-time comparison (so an attacker can't time-attack your check):

Verification
node.js 
const crypto = require('crypto');

function verifySignature(payload, signature, secret) {
  const expected = crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expected)
  );
}

app.post('/webhook', (req, res) => {
  const signature = req.headers['x-Elevate-signature'];
  const raw = JSON.stringify(req.body);

  if (!verifySignature(raw, signature, WEBHOOK_SECRET)) {
    return res.status(401).json({ error: 'Invalid signature' });
  }

  res.status(200).json({ received: true });
});

Heads up: JSON.stringify(req.body) only works if your framework hasn't reformatted the body. If you're using Express with express.json(), switch to express.raw({ type: 'application/json' }) for your webhook route specifically and pass req.body (a Buffer) directly to the HMAC. Same idea applies in Flask, Rails, ASP.NET — read the raw bytes, not the parsed object.

Always verify before you do anything.If the signature check fails, return 401 and stop — don't log the body, don't enqueue the event, don't ping anyone. An unverified payload is hostile by default.

Testing

Before you trust a webhook in production, send a test ping. This fires a synthetic ping event at your endpoint with a real signature, so you can verify your handler accepts it without waiting for someone to actually move a task in the app:

POST/workspaces/{wid}/webhooks/{whid}/test
curl -X POST https://app.arcnvd.com/api/v1/workspaces/trust-and-safety/webhooks/wh_abc123/test \
  -H "X-API-Key: Elevate_abc123def456"

If the call returns 200 on our side and your endpoint logs receipt of a ping event, you're set. If you don't see anything land, the most common culprits are: signature verification failing on the test, a firewall blocking inbound from Elevate, or your endpoint requiring auth that the webhook can't satisfy. Local development tools like ngrok or Cloudflare Tunnel are the easiest way to test from your laptop.

Managing webhooks

Listing webhooks shows everything attached to a workspace, which is handy when you're cleaning up old test endpoints or auditing what's subscribed to your data:

GET/workspaces/{id}/webhooks
curl https://app.arcnvd.com/api/v1/workspaces/trust-and-safety/webhooks \
  -H "X-API-Key: Elevate_abc123def456"

If you need to pause a webhook temporarily (deploying a new version of your handler, debugging an issue) you don't have to delete and recreate it. Set active to false with PATCH and Elevate will stop sending events to that URL until you flip it back:

PATCH/workspaces/{wid}/webhooks/{whid}
curl -X PATCH https://app.arcnvd.com/api/v1/workspaces/trust-and-safety/webhooks/wh_abc123 \
  -H "X-API-Key: Elevate_abc123def456" \
  -H "Content-Type: application/json" \
  -d '{"active":false}'

Pausing also keeps the secret stable, so you don't have to redistribute new credentials when you bring it back online.

Retry policy

Networks fail, services restart, deploys happen. When your endpoint returns anything other than a 2xx status — or doesn't respond at all — Elevate retries the delivery up to 3 times with exponential backoff at 5 seconds, 30 seconds, and 5 minutes. If all three retries fail, the event gets dropped and we move on.

The X-Elevate-Delivery header stays the same across all retry attempts for a given event. If you're processing webhooks asynchronously (writing them to a queue, for example), use that ID as your idempotency key so a successful delivery on retry #2 doesn't re-trigger the side effects you already ran on retry #1.

For high-stakes integrations (billing, audit trails) we recommend pulling state via the API in a periodic reconciliation job rather than relying on webhook delivery alone. Webhooks are best-effort; the API is authoritative.
