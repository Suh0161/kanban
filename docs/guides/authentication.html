Authentication

Elevate gives you two ways to authenticate: JWT bearer tokens for human users in a browser, and API keys for everything else — bots, CI pipelines, integrations, scripts. Both methods unlock the same endpoints, so the choice is really about how the credential is created and rotated.

JWT bearer tokens

JWTs are what the Elevate web app uses behind the scenes. When a user signs in, the backend hands back a signed token that the frontend tucks into localStorage and replays on every API call. They're great for short-lived sessions tied to a specific human, and they automatically expire — which is exactly what you want for browser code.

Obtain a token

Log in with the user's email and password. The response includes both the user record and a fresh JWT:

POST/auth/login
curl -X POST http://localhost:3001/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com","password":"your-password"}'

Tokens stay valid for seven days. After that they're rejected with a 401 and you'll need to log in again. The web app handles this automatically by redirecting back to the login page on any 401 response.

Use the token

Send it in the Authorization header on every authenticated request, prefixed with Bearer :

GET/auth/me
curl http://localhost:3001/api/v1/auth/me \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."
Use JWTs for human sessions, not for bots.JWTs expire and have to be refreshed. If you're building a bot or a long-running integration, jump down to API keys — they're built for that use case and don't need rotation.

API keys

API keys are the right choice for anything that isn't a person sitting at a browser. CI pipelines, monitoring scripts, Slack bots, deploy hooks — any code that needs to talk to Elevate without a human in the loop. They don't expire by default (you can set an expiry if you want), and each key carries a scope that limits what it can do.

Generate a key

Open Settings → API Keys in the Elevate app and click Generate API Key. Give it a name that tells future-you what it's for ("github-actions", "datadog-bot", "weekly-report-script") and pick a scope:

  • Read & Write — full access to every endpoint. Use for tools that create or modify tasks.
  • Read Only — restricted to GET requests. Perfect for dashboards, exporters, and analytics jobs that should never accidentally write.

The raw key value is shown once at creation time and never again. Copy it immediately into your secrets manager. Every key starts with Elevate_ so it's easy to spot in logs and config files.

Use a key

Send it in the X-API-Key header — that's the recommended way for almost every situation:

GET/board/{workspaceId}
curl http://localhost:3001/api/v1/board/trust-and-safety \
  -H "X-API-Key: Elevate_abc123def456"

Header-only API keys

API keys are only accepted in the X-API-Key header. The API rejects api_key query parameters so credentials do not end up in access logs, browser history, shared screenshots, or reverse proxy logs.

Do not put keys in URLs. If a client cannot send custom headers, proxy the request through a component that can add X-API-Key server-side.

Managing API keys

You can list, create, and revoke keys through the same API the dashboard uses. Listing your keys is helpful when you're auditing what's out there or building a self-service rotation tool:

GET/workspaces/{id}/api-keys
curl http://localhost:3001/api/v1/workspaces/trust-and-safety/api-keys \
  -H "X-API-Key: Elevate_abc123def456"

Note that the response only ever returns the key prefix and metadata — the raw key value isn't stored anywhere we could give back. If a key leaks, revoke it with DELETE and create a new one. Revocations take effect immediately.

Error responses

When authentication fails — missing header, expired token, revoked key, malformed credentials — the API returns 401 Unauthorized with a structured body:

401Unauthorized
json 
{
  "error": "Invalid or missing API key",
  "code": "UNAUTHORIZED"
}

Always check for 401 specifically and trigger a re-auth flow rather than retrying with the same credential. Hammering on a bad key won't make it good and will eventually trip the rate limiter.
